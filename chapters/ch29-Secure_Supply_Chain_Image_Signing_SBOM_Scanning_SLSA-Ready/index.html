    üß± PART 1 ‚Äî Install the Core Tools¶

    Install Cosign (image signing)¶

    brew install cosign

    or:

    curl -sSL https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64 -o cosign
chmod +x cosign

    Install Syft (SBOM)¶

    brew install syft

    Install Grype (vulnerability scanning)¶

    brew install grype

    Install Trivy (full security scanner)¶

    brew install trivy

    Now you're fully equipped ‚úîÔ∏è

    üß± PART 2 ‚Äî Generate an SBOM for Your Image¶

    SBOM = Software Bill Of Materials It lists everything inside your image:

    • OS packages
    • libraries
    • dependencies
    • versions
    • licenses

    Generate:

    syft my-image:latest -o json > sbom.json

    Or human-readable:

    syft my-image:latest

    This is REQUIRED for compliance (NIST, SLSA, EU CRA).

    üß™ PART 3 ‚Äî Scan Image for Vulnerabilities¶

    Using Grype:

    grype my-image:latest

    Using Trivy (better for full pipeline):

    trivy image my-image:latest

    Trivy also finds:

    ‚úîÔ∏è vulnerabilities ‚úîÔ∏è misconfigurations ‚úîÔ∏è secrets ‚úîÔ∏è license issues ‚úîÔ∏è SBOM components

    Goal: block images with critical vulnerabilities.

    Example Trivy output:

    CRITICAL: openssl vulnerability CVE-2023-xxxx

    Fix before deploying.

    üß± PART 4 ‚Äî Image Signing with Cosign (Sigstore)¶

    Sign your image:

    cosign generate-key-pair

    This creates:

    • cosign.key
    • cosign.pub

    Sign the Docker image:¶

    cosign sign -key cosign.key my-image:latest

    This attaches a cryptographic signature TO THE IMAGE, stored in the OCI registry.

    üîç PART 5 ‚Äî Verify the Signature¶

    Verify:

    cosign verify -key cosign.pub my-image:latest

    Output:

    ‚úîÔ∏è Valid signature ‚úîÔ∏è Identity of signer ‚úîÔ∏è Certificate chain

    If signature is missing or invalid ‚Üí REJECT.

    üîê PART 6 ‚Äî Enforce Image Signing in Kubernetes¶

    We use Sigstore Policy Controller (formerly Cosigned).

    Install:

    kubectl apply -f https://github.com/sigstore/policy-controller/releases/latest/download/policy-controller.yaml

    Now create a policy:

    apiVersion: policy.sigstore.dev/v1beta1
kind: ClusterImagePolicy
metadata:
  name: require-signed-images
spec:
  images:
    - glob: "ghcr.io/myorg/*"
  authorities:
    - key:
        data: |
          -----BEGIN PUBLIC KEY-----
          ...your cosign.pub...
          -----END PUBLIC KEY-----

    This says:

    ‚úîÔ∏è Only images signed by YOU can run ‚úîÔ∏è Unsigned or tampered images are blocked ‚úîÔ∏è Protects your production cluster

    Try deploying an unsigned image:

    ‚ùå AdmissionWebhook DENIES the deployment ‚úîÔ∏è Perfect protection

    üß± PART 7 ‚Äî Secure Supply Chain in CI/CD¶

    A real pipeline includes:

    Step 1 ‚Äî Build¶

    Create minimal, multi-stage images.

    Step 2 ‚Äî Scan image¶

    Block critical vulnerabilities:

    trivy image --exit-code 1 my-image:latest

    Step 3 ‚Äî Generate SBOM¶

    syft . -o cyclonedx-json > sbom.json

    Step 4 ‚Äî Sign the image¶

    cosign sign -key cosign.key my-image:latest

    Step 5 ‚Äî Push SBOM to registry¶

    Attach SBOM to image:

    cosign attach sbom -sbom sbom.json my-image:latest

    Step 6 ‚Äî Deploy via ArgoCD¶

    ArgoCD only applies manifests containing signed & verified images.

    This is SLSA Level 2 / Level 3 ready.

    üîí PART 8 ‚Äî Prevent Running Vulnerable Images (OPA Gatekeeper)¶

    Policy example:

    apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sVulnScan
metadata:
  name: block-critical-images
spec:
  match:
    kinds:
      - apiGroups: [""]
        kinds: ["Pod"]
  parameters:
    severity: ["critical"]

    This prevents:

    ‚ùå images with CRITICAL CVEs ‚ùå images missing SBOM ‚ùå images without valid signature

    Zero-trust for container images.
