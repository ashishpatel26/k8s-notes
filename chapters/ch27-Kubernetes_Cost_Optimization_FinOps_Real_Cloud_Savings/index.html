    ‚≠ê Why Kubernetes Becomes Expensive¶

    üß± PART 1 ‚Äî Resource Requests & Limits (HUGE SAVINGS)¶

    Most companies waste money because Pods ask for too much CPU and too much memory.

    Example wasteful Deployment:

    resources:
  requests:
    cpu: "2"
    memory: "4Gi"

    But real usage may be:

    • CPU: 300m
    • Memory: 700Mi

    This is 6x waste.

    FIX: Right-Sizing¶

    Use Vertical Pod Autoscaler (VPA) recommendations:

    kubectl describe vpa backend-vpa

    Then adjust Deployment.

    üß± PART 2 ‚Äî Use Cluster Autoscaler (CA)¶

    Cluster Autoscaler automatically:

    ‚úîÔ∏è removes empty nodes ‚úîÔ∏è adds nodes during load ‚úîÔ∏è shrinks the cluster at night

    EKS example:

    eksctl utils associate-iam-oidc-provider --cluster mycluster
eksctl create nodegroup \
  --cluster mycluster \
  --asg-access \
  --nodes 3 \
  --nodes-min 1 \
  --nodes-max 10

    ‚úîÔ∏è Only pay for needed nodes ‚úîÔ∏è Zero idle capacity

    This can save 30‚Äì50% ALONE.

    üß± PART 3 ‚Äî Use SPOT / PREEMPTIBLE Nodes (Massive Savings)¶

    Spot nodes cost:

    • ‚ùó 70‚Äì90% cheaper than on-demand
    • Perfect for stateless workloads

    Add a spot-only node pool:

    AWS example (eksctl):

    eksctl create nodegroup \
  --cluster mycluster \
  --name spot-nodes \
  --instance-types t3.medium \
  --spot \
  --nodes 2 \
  --nodes-min 0 \
  --nodes-max 20

    Then label:

    kubectl label node <spot-node> spot=true

    Deploy cheap workloads to it:

    nodeSelector:
  spot: "true"

    Huge savings.

    üß± PART 4 ‚Äî Use Efficient Instance Types¶

    Bad choice examples:

    ‚ùå cpu-heavy nodes for memory apps ‚ùå small nodes for giant workloads causing fragmentation ‚ùå expensive ‚Äúburstable‚Äù nodes with no need

    General rule:

    ‚úîÔ∏è CPU-heavy apps ‚Üí compute-optimized ‚úîÔ∏è Memory-heavy apps ‚Üí memory-optimized ‚úîÔ∏è Mixed ‚Üí general-purpose

    Correct instance types reduce hidden waste.

    üß± PART 5 ‚Äî Reduce Log & Metrics Costs¶

    A BIG SECRET: Logging & monitoring often costs more than compute.

    Optimize:¶

    ‚úîÔ∏è Use Loki instead of Elasticsearch ‚úîÔ∏è Drop DEBUG logs in production ‚úîÔ∏è Short retention (3‚Äì7 days) ‚úîÔ∏è Avoid shipping k8s events to logs ‚úîÔ∏è Only collect necessary namespace logs

    ‚úîÔ∏è Downsample ‚úîÔ∏è Drop high-cardinality metrics ‚úîÔ∏è Reduce scrape intervals

    Large images = slower scaling + wasted storage.

    ‚úîÔ∏è Use Alpine ‚úîÔ∏è Multi-stage builds ‚úîÔ∏è Use distroless images ‚úîÔ∏è Remove unused libraries ‚úîÔ∏è Compress layers

    üß± PART 7 ‚Äî Autoscaling Improvements (HPA + KEDA)¶

    HPA¶

    ‚úîÔ∏è scale by CPU/memory ‚úîÔ∏è core autoscaling

    ‚úîÔ∏è scale by queue length ‚úîÔ∏è HTTP traffic ‚úîÔ∏è Kafka lag ‚úîÔ∏è Prometheus queries

    ‚úîÔ∏è Right-sizing pod requests ‚úîÔ∏è Using VPA recommendations ‚úîÔ∏è Using bin-packing scheduling strategies ‚úîÔ∏è Using larger nodes (often cheaper per CPU/mem)

    üß± PART 9 ‚Äî Use ‚ÄúRequests‚Äù Correctly (Most Misunderstood)¶

    Requests determine how much CPU/mem the scheduler allocates.

    Best practice:

    ‚úîÔ∏è Use requests, optional or small limits ‚úîÔ∏è Use VPA to auto-tune requests ‚úîÔ∏è Avoid high memory limits (OOM kills your app) ‚úîÔ∏è Avoid high CPU requests (prevents bin-packing)
